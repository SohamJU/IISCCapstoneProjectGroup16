# Synthetic Data Pipeline Architecture

## Overview
The synthetic data pipeline (`pipelines/02_run_synthetic_data_pipeline.py`) is the core orchestration engine of the data layer. It transforms raw, disjointed product and review datasets into a massive, highly-relational, and perfectly validated synthetic eCommerce database. 

It handles the entire ETL (Extract, Transform, Load) lifecycle: from downloading real-world Amazon data from Hugging Face, to injecting synthetic customers, orders, and returns, and finally pushing the verified relational tables to a cloud PostgreSQL database.

---

## Pipeline Architecture & Flow

The pipeline is strictly sequential and divided into **13 discrete steps**. To ensure idempotency and speed, the pipeline automatically detects if a step has already been completed and skips it unless explicitly forced.

### Phase 1: Ingestion & Preprocessing (Steps 1-4)
* **Step 1: Download Amazon Reviews 2023**: Connects to Hugging Face datasets and streams raw product metadata and reviews (e.g., Electronics, Appliances), saving them as chunked Parquet files.
* **Step 2: Preprocess Product Catalog**: Cleans the raw product metadata, extracts primary categories, cleans HTML from descriptions, and exports a clean `product_catalog.csv`.
* **Step 3: Preprocess Reviews**: Cleans the raw reviews, dropping missing user IDs and filtering for verified purchases to generate `reviews.csv`.
* **Step 4: Kaggle Compatibility**: Ensures the processed schema remains compatible with the legacy Kaggle dataset formats to prevent downstream breaking changes.

### Phase 2: Synthetic Generation (Steps 5-9)
* **Step 5: Generate Customers**: Uses realistic distribution models (Faker) to generate `customers.csv` mapping directly to the distinct `user_id`s found in the real reviews.
* **Step 6: Generate Orders & Items**: Simulates historical purchase behavior. It creates a `orders.csv` table and a relational `order_items.csv` table linking customers to the exact products they reviewed.
* **Step 7: Generate Returns**: Synthesizes a `returns.csv` table. It models a realistic 5-15% return rate on delivered orders, ensuring logical date constraints (return date > delivery date).
* **Step 8: Generate Queries**: Runs both generic and enhanced query generators, then merges the results into a single `customer_queries.csv`, combining generic support questions with order-aware realistic tickets.
* **Step 9: Generate Policy Documents**: Synthesizes vector-ready Markdown knowledge base files representing store policies (Returns, Shipping, etc.).

### Phase 3: Validation & Publishing (Steps 10-13)
* **Step 10: Integrity Validation**: Runs a rigorous suite of 13 programmatic validation checks. It verifies foreign key constraints (e.g., *does every return match a real order item?*), mathematical proofs (e.g., *does qty * price = total?*), and timeline logic.
* **Step 11: Schema Generation**: Automatically analyzes the generated CSVs and produces highly-detailed `.schema.json` files for agents to consume, complete with data types and automatically extracted unique values for categorical columns.
* **Step 12: PostgreSQL Upload**: Connects to the shared Aiven Cloud PostgreSQL instance and uploads all 7 datasets into perfectly mapped relational tables.
* **Step 13: PostgreSQL Diagnostics**: Runs SQL verification queries (Row Counts, Data Fetches, and Relational JOINs) directly against the cloud database to guarantee cloud data integrity.

## How to Use It

The script exposes a powerful Command Line Interface (CLI). You can run it via the source module or the notebook wrapper:

### 1. Run the Full Pipeline
To run the entire pipeline end-to-end (it will smartly skip steps that are already completed):
```bash
python pipelines/02_run_synthetic_data_pipeline.py
```

### 2. Force Re-run Everything
If you want to completely overwrite the existing data and start from scratch, use the `--force` flag:
```bash
python pipelines/02_run_synthetic_data_pipeline.py --force
```

### 3. Run a Specific Step
If you only want to execute one specific step (for example, just testing the PostgreSQL upload without regenerating orders), use `--step`:
```bash
# Run only Step 11 (Generate Schemas)
python pipelines/02_run_synthetic_data_pipeline.py --step 11 --force

# Run only Step 13 (Test Database)
python pipelines/02_run_synthetic_data_pipeline.py --step 13
```

### 4. Customize Synthetic Scaling
You can control the volume of the synthetic data generated by passing scale arguments:
```bash
python pipelines/02_run_synthetic_data_pipeline.py --num-customers 5000 --num-orders 20000 --total-queries 1000
```

### 5. Control PostgreSQL Upload Behavior
By default, the PostgreSQL upload behavior is determined by the config. You can override this on the fly:
```bash
# Force the pipeline to append to existing cloud tables instead of replacing them
python pipelines/02_run_synthetic_data_pipeline.py --step 12 --postgres-behavior append
```

## Configuration & Environment Variables

The pipeline behavior is heavily driven by variables located in `src/config/data.py` and your local `.env` file. Here are the key variables and the effect of modifying them:

### Environment Variables (`.env`)
* **`POSTGRESQL_AIVEN_PASSWORD`**: The master password for the cloud database. **Effect**: If this is missing or incorrect, Steps 12 & 13 will safely skip to prevent crashing, but data will not be uploaded.
* **`POSTGRESQL_HOST`, `POSTGRESQL_PORT`, `POSTGRESQL_USER`, `POSTGRESQL_DB`**: Database connection endpoints. **Effect**: They default to the shared group Aiven instance. If you override these in `.env`, the pipeline will upload data to your own custom PostgreSQL database instead.

### Data Scaling & Generation Variables (`src/config/data.py`)
* **`AMAZON_2023_MAX_REVIEWS` (Default: `10,000`)**: Caps the number of raw reviews downloaded per product category for the proof-of-concept. **Effect**: Increasing this pulls significantly more real-world data from Hugging Face but increases processing time and file sizes.
* **`AMAZON_2023_CATEGORIES` (Default: `["Electronics", "Appliances"]`)**: The product categories to ingest. **Effect**: Adding categories like `"Toys_and_Games"` expands the product catalog and diversity of reviews.
* **`NUM_CUSTOMERS` (Default: `2,000`)**, **`NUM_ORDERS` (Default: `10,000`)**, **`NUM_RETURNS` (Default: `2,000`)**: Targets for the synthetic generators. **Effect**: Changing these determines exactly how large your synthesized tables will be. Scaling up makes queries more realistic for big-data tests, but generation takes longer.

### Upload Behavior
* **`POSTGRESQL_UPLOAD_BEHAVIOR` (Default: `"skip"`)**: Determines how Step 12 interacts with existing tables.
  * **`replace`**: Drops the existing tables entirely and uploads a fresh copy (best for clean testing).
  * **`append`**: Leaves existing data intact and inserts the newly generated rows at the bottom (useful if you are generating data in batches).
  * **`skip`**: Skips the database upload phase completely (useful for local-only testing).
